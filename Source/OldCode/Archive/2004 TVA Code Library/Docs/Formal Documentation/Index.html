Using the Code on Your Development System

 

To use the shared code library, reference the latest version in the following folder:

 

\\opdat\opdat\CtrlSys\CodeLibrary\Builds\Release\Current

 

You can then reference the desired libraries in your projects and start using the code.  Note that many of the assemblies contain designable components that can be added to your component list in Visual Studio.  The compiled assemblies found in the current release build folder will always be tested and considered reliable.  The absolute latest build can be found in the beta build folder \\opdat\opdat\CtrlSys\CodeLibrary\Builds\Beta  however, this will likely be untested code and you should only use this code if you absolutely need a need a feature or bug fix that this code provides.

* If you are using Visual Studio 2003 you can ignore this note. Because the TVA compression library uses some unmanaged (i.e., not .NET garbage collected) C code (zlib source code is used for compression and standard zip file manipulation), use of this class as a designable component forces the following limitations:

 

1)       You must reference the assembly in your project before dragging it onto a design surface

2)       You must change the Copy Local property of the reference to False

3)       You must manually copy the assembly into your project's bin folder so it can find the assembly at run time

4)       After performing these steps you can safely drag the component onto a design surface

 

This is only a limitation that is incurred under Visual Studio 2002 - this is not an issue when using Visual Studio 2003.  Visual Studio 2002 apparently only loads assemblies as a byte stream and this is not possible if an assembly contains unmanaged code.

    Imports TVA.Shared.DateTime

Public Shared Function GetWin32TimeZone(ByVal StandardName As String) As Win32TimeZone

HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Time Zones

 

Note that it doesnt matter if the incoming date is in daylight savings time or standard time in order to properly perform the conversion, however, there will be a couple of hours during fall back when the conversion will be ambiguous  for this reason I recommend we *always* store dates in UTC whenever possible; but when we cant, these functions will come in handy.

New TVA Code Library Test Bed

 

Based on the testing architecture Steve Lowe had put together, we now have a standard test bed for unit-testing the code library:

 

 

The test bed application is located at \\opdat\opdat\CtrlSys\CodeLibrary\Testing - open the TestBed solution to see the code.  New unit tests can be easily added here by creating class libraries that derive from TVA.Testing.TestBase and implement the desired tests.  All methods in TestBase derived classes that have a name that begins with Test and have no parameters will be executed.  Note that tests should be designed so that they can be run in random order.  Please volunteer any free time you have into creating a good set of test cases for existing library code, Steven suggests the following:

 

There are basically four kinds of tests; all are important, but in my opinion the first two are the most critical as bugs in these areas can be subtle and can go undetected for quite a while -

 

1. range tests - does the component/method behave correctly with a range of valid inputs? special cases for range extremes (like zero, nothing, max value, empty value, etc.) and typical cases are appropriate here

 

2. failure tests - does the component/method behave correctly with inputs that would be expected to fail? for example, a file method asked to write to a file that does not exist or which is write-protected may be expected to fail quietly or throw a specific exception

 

3. Path tests or end-to-end tests - tracking through the lifecycle of an entity/component/object - are important to satisfy/test major use-cases.

 

4. stress tests - does the component/method continue to behave correctly under heavy usage? Stress testing is limited within this framework due to its single-machine nature; Rational test suite is more appropriate for true stress tests, but there is merit in running a few threads doing things to a component, even if they're doing them at random - if for no other reason than to show that the component is thread-safe

Web Crawler Component Documentation

 

Semantic Rules / Functional Notes:

 

Only indexing directives appearing inside of an HTML <HEAD> tag will be considered:

 

<HTML>

            <HEAD>

                  IndexingDirectives

            </HEAD>



 

All tag, attributes and content are case and order insensitive.

 

Only the INDEXME and FOLLOWLINKS directives are considered page-level and are targeted for indexing control of the current page.  All other directives are just for general control of the web-crawler.

 

Multiple indexing directives are allowed to be specified, for example:

 

      <META NAME=ESOINDEX INDEXME=TRUE>

      <META NAME=ESOINDEX FOLLOWLINKS=FALSE>

      <META NAME=ESOINDEX ALSOINDEX=http://www.yahoo.com>

      <META NAME=ESOINDEX ALSOINDEX=SiteIndex.asp?DisplayType=Alpha>

 

is the same as the following:

 

      <META NAME=ESOINDEX ALSOINDEX=http://www.yahoo.com ALSOINDEX=SiteIndex.asp?DisplayType=Alpha INDEXME=TRUE FOLLOWLINKS=FALSE>

 

Only multiple occurrences of the ALSOINDEX directive will be respected.  When multiple tags (or attributes) exist that specify a previously defined directive, the first encountered directive will take precedence.

 

File system indexing will only occur when both the FILESYSTEMROOT and FILEWEBROOT directives exist.  These directives are intended to direct the web-crawler to index files from a file system perspective when these files are already assessable from the web (this is useful for indexing files that are available to be served by the web server, yet may or may not have direct links to these files  like in the case of the ESO ad-hoc document library).

 

The optional FILEINCLUSION, FILEEXCLUSION, DIRRECURSE, DIRINCLUSION, and DIREXCLUSION directives allowed detailed control of a file-system based index.  These directives are ignored unless the FILESYSTEMROOT and FILEWEBROOT directives exist.

 

The optional DIRINCLUSION and DIREXCLUSION directives are ignored unless the DIRRECURSE is TRUE.

 

The ALSOINDEX directive allows additional pages to be indexed by the web-crawler; the specified page will be indexed even if the page is not in the web-crawlers web site restriction list.  Any INDEXME or ALSOINDEX directives on the destination page will still be respected, however, no general link and/or frame traversion will occur for pages being indexed outside the web-crawlers web site restriction list.
