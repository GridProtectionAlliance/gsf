Why have a common code library?

 

Standards

          Standards in development make projects easier to maintain.  The developer of an application may not be the same person who ends up maintaining the code.

Speed

       The component model says: “It’s faster not to reinvent the wheel” (not to mention cheaper)

Reliability

       Reusing battle hardened tested code will produce more reliable applications.  This is critical in this type of environment.

Where do I get the library?

 

Reference binaries directly from \\opgfs\opdat\ in the "\\opgfs\opdat\CtrlSys\CodeLibrary\2.0\Builds\Release\Current\" folder – this will make sure your code is always referencing the very latest production build.  If you think something in the latest build has caused your code to break, the old builds will *always* be available in their respective folder.

 

How do I start using the library?

 

          •  Add designable components to your Visual Studio toolbox and start dragging desired components onto design surfaces.

•  Reference desired assemblies in your projects directly from  "\\opgfs\opdat\CtrlSys\CodeLibrary\2.0\Builds\Release\Current\" and add the needed Imports statements to your code (see examples).

 

How do I add code to the library?

 

·        Send component description to the ESO Dot Net Reuse Committee (DNRC) e-mail group

·        If your code is approved for inclusion, the DNRC will give you a namespace assignment for your code or component

·        Place your code or component in the given namespace and rerun the test script to make sure your code still works after this change

·        Send code and test script to DNRC for library integration

·        That’s it!

How do I add custom settings to my app.config file?

 

Imports Tva.Configuration.Common

 

1) Using DefaultConfigFile() property

 

            ' Going through the DefaultConfigFile() shared property will give access to .Net's

            ' <appSettings> and <connectionStrings> sections.

            With DefaultConfigFile()

                ' Adds settings to the <appSettings> section.

                .AppSettings.Settings.Add("SaveSettingsOnExit", "1")

 

                ' Adds settings to the <connectionStrings> section.

                .ConnectionStrings.ConnectionStrings.Add(New ConnectionStringSettings("DbConnectString", "Server=RGOCSQLD;Database=DB;Trusted_Connection=True"))

 

                ' Adds settings to the <categorizedSettings> section.

                .CategorizedSettings.General.Add("SaveSettingsOnExit", "1")

                .CategorizedSettings("Development").Add("DbConnectString", "Server=RGOCSQLD;Database=DB;Trusted_Connection=True", True)

                .CategorizedSettings("Production").Add("DbConnectString", "Server=OPDATSQL;Database=DB;Trusted_Connection=True", True)

 

                ' Saves modified settings to the config file.

                .Save()

            End With

 

2) Without using DefaultConfigFile() property

 

            ' Adds settings to the <categorizedSettings> section.

            Settings.Add("SaveSettingsOnExit", "1") ' This will add the setting in "general" category.

            CategorizedSettings("Development").Add("DbConnectString", "Server=RGOCSQLD;Database=DB;Trusted_Connection=True", True)

            CategorizedSettings("Production").Add("DbConnectString", "Server=OPDATSQL;Database=DB;Trusted_Connection=True", True)

 

            ' Saves modified settings to the config file.

            SaveSettings()

 

3) Retrieving settings

 

            ' Retrieve settings from the config file.

            Debug.WriteLine(Settings("SaveSettingsOnExit").Value())

            Debug.WriteLine(CategorizedSettings("Development")("DbConnectString").Value())

            Debug.WriteLine(CategorizedSettings("Production")("DbConnectString").Value())
